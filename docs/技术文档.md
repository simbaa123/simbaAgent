# simbaAgent 技术文档

本文基于当前仓库实现，系统说明：需求分析、设计思路、技术选型与具体实现。项目定位为“可演示的电商售后智能客服工作台”，重点展示 **流式 Agent 输出 + 可观测工具链路 + 人工确认闭环（HITL）+ MCP 扩展能力**。

## 1. 需求分析

### 1.1 业务目标（What）

- 提供一个售后工作台，支持客服/坐席在多个会话间切换，并对单会话进行处理。
- 覆盖典型售后诉求的演示闭环：
  - 查物流（按订单号/手机号后四位定位订单 → 查询轨迹 → 输出建议）
  - 退货（可退判断 → 二次确认 → 创建退货申请）
  - 改地址（已支付未发货可改 → 二次确认 → 修改地址成功回执）
  - 政策问答（RAG 演示：检索知识库条款 → “引用卡片 + 文本回答”）
- 输出不仅是“答案”，还要输出 **过程**（计划、工具调用、工具结果、确认弹窗、导出/审计结果）。

### 1.2 用户与场景（Who）

- 用户：售后客服/运营同学/面试官（演示对象）。
- 场景：从 Inbox 进入 Console，输入问题，看到 Agent 逐步推进（Plan）、调用工具（日志）、逐字生成回复（assistant_delta）。

### 1.3 关键非功能需求（How well）

- 流式体验：回答与过程事件实时到达，前端不需要轮询。
- 可观测与可解释：工具调用必须可视化，避免“黑盒回答”。
- 可交互安全：涉及敏感动作（退货创建/改地址）必须经过用户确认。
- 可扩展：未来可替换真实系统（订单/物流/KB）或接入 MCP 工具，不重写主链路。
- 可演示稳定：即使首次加载/编译导致慢，也需要尽快返回“首包事件”，避免前端一直等待。

## 2. 设计思路

### 2.1 总体架构

- 前端：Vue 3 工作台（Inbox/Console）。
- 后端：Express 提供 REST（列表/详情）+ SSE（流式 chat）。
- Agent：LangChain 负责模型调用与工具编排；业务工具用于查询/动作；MCP 工具用于导出与 SQLite 审计。

端到端链路（简化）：

1) 用户在 Console 输入 → 前端 `fetch POST /api/chat/stream`
2) 后端建立 SSE 流，先推送 `plan_update`（启动/初始化），并定时 `ping`
3) 后端运行 `runLangChainAgent`
4) Agent 根据意图走脚本化分支（物流/退货/改地址/政策）或走 LLM+Tools（其他）
5) 后端持续推送事件：
   - `assistant_delta`：增量文本
   - `tool_call/tool_result`：工具链路
   - `need_choice/need_confirm`：人工选择/确认
   - `export_ready/sqlite_result`：MCP 扩展结果
6) 结束时发送 `final`

### 2.2 为什么用 SSE（而不是 WebSocket / EventSource）

- SSE 适合本项目：主方向是“服务端 → 前端”的持续推送，协议简单、部署成本低。
- 与 WebSocket 相比：无需维持双向复杂状态；用户交互通过下一次 HTTP 请求即可补足。
- 与 EventSource 相比：本项目需要 `POST` 且带 body（conversationId/message/context），EventSource 只支持 GET，因此采用 `fetch + ReadableStream` 自解析 SSE。

### 2.3 可观测性：把“文本流”与“过程流”拆开

- 文本流：`assistant_delta`（token/分块增量），用于 UI 的“逐字输出”。
- 过程流：`plan_update/tool_call/tool_result/...`，用于侧栏展示“Agent 正在做什么”。
- 价值：面试/演示时能清楚说明“为什么这么答”，也便于 debug。

### 2.4 Human-in-the-loop：敏感动作必须确认

- 退货创建与改地址都采用：
  - 先判断是否可执行（可退/可改）
  - 推送 `need_confirm`（携带 action + details）
  - 前端弹窗确认后，再发起下一次 `/api/chat/stream`，把 `context.confirm` 回传
  - 后端执行工具并输出结果

这使得“模型不直接落敏感操作”，同时保持了演示闭环。

### 2.5 可扩展性：业务能力以 Tools 形式插件化

- 查询类：searchOrders/getOrderDetail/getShipmentTracking/kbSearch
- 操作类：createReturnRequest/modifyOrderAddress
- 扩展类（MCP）：exportConversationToFile/sqliteQuery

Tools 的核心原则：

- 执行业务逻辑
- 同步发送 `tool_call/tool_result`，对前端透明

## 3. 技术选型

### 3.1 前端

- Vue 3：组件化清晰，适合工作台 UI。
- Vite：开发启动快，代理 /api 到后端方便。
- Tailwind：快速搭建 UI（列表、卡片、弹窗）。

### 3.2 后端

- Node.js + Express：提供 REST + SSE，落地简单。
- tsx：直接运行 TS，减少编译步骤，适合快速迭代与演示。

### 3.3 Agent / LLM

- LangChain：标准化 agent + tools 接口。
- `@langchain/openai` 的 ChatOpenAI：同时兼容 OpenAI 与 DeepSeek（通过 baseURL）。
- 流式：开启 `streaming: true`，把模型 delta 转成 SSE `assistant_delta`。

### 3.4 MCP

- filesystem：用于导出会话到 exports 目录。
- sqlite：用于审计记录与查询回放。

## 4. 具体实现

### 4.1 后端入口：/api/chat/stream

文件：`server/index.ts`

- 解析 body（conversationId/message/context）
- 初始化 SSE：写入 headers，立即发 `plan_update(boot)` 作为首包，并启动 `ping` 心跳
- 记录用户消息：写入内存消息列表（用于后续 /export）
- 动态 import 并运行 Agent：`runLangChainAgent({res, conversationId, userMessage, traceId, context})`
- finally：发送 `final` 并 `res.end()`

设计要点：

- **首包事件**：解决“连接建立但前端一直等不到第一段数据”的体验问题。
- **心跳**：避免中间网络/代理将连接误判为空闲。

### 4.2 SSE 协议与事件

文件：`server/sse.ts`

- `initSse(res)`：设置 `text/event-stream`、`keep-alive`、`no-cache` 等。
- `sendSseEvent(res, event, data)`：写入 `event:`/`data:` 帧，使用空行分隔。
- `ssePing(res)`：发送轻量心跳。

事件约定（核心）：

- `plan_update`：更新右侧 Plan（steps 数组）
- `assistant_delta`：增量输出
- `tool_call/tool_result`：工具链路
- `need_choice/need_confirm`：选择/确认弹窗
- `export_ready/sqlite_result`：MCP 结果
- `error/final`：错误与结束

### 4.3 前端流式读取：fetch + ReadableStream

文件：`src/ui/pages/ConsolePage.vue`

- 使用 `fetch` 发送 `POST /api/chat/stream`
- 使用 `res.body.getReader()` 逐块读取
- 用 `TextDecoder` 增量解码拼接到 buffer
- 调用 `parseSseTextChunk` 将 buffer 解析为事件数组

解析器：`src/ui/pages/console/sse.js`

- 处理 “网络分片导致半包/粘包” 的问题
- 用 `\n\n` 作为事件分隔符，解析 event/data

### 4.4 Agent 主体：意图 → Plan → Tools / LLM

文件：`server/langchainAgent.ts`

关键模块：

- `createModel()`：OpenAI/DeepSeek 兼容模型工厂，开启 streaming。
- `createPlanSteps(intent)`：为前端 Plan 生成骨架。
- `buildPriorMessages`：从历史消息构造短期记忆。
- `createToolWrapper`：统一封装 tool_call/tool_result 事件派发，减少重复代码。

编排入口：`runLangChainAgent`

- 特殊命令：/export、/sql、/audit 直接走 MCP 处理
- HITL 回流：context.confirm.action 匹配后执行 createReturnRequest/modifyOrderAddress
- 意图分支：logistics/return/modify_address/policy
- other：走 LangChain Agent + Tools，并把 streamEvents 转成 assistant_delta

### 4.5 数据与导出

文件：`server/dataStore.ts`

- `loadSampleData()` 读取本地演示数据并缓存
- 启动时清空 `messages`，保证会话默认从空白开始
- `appendMessage()` 把用户与 agent 的最新消息写入内存
- /export 导出时使用内存中的 messages，保证导出包含真实对话

## 5. 运行与演示建议

- 启动：`npm run dev`
- 进入：Inbox → Console
- 典型演示脚本：
  - “我手机尾号3344，帮我查物流”
  - “我要退货（质量问题）” → 弹窗确认 → 创建退货
  - “我要改地址，新地址：广东省深圳市南山区xxx” → 弹窗确认 → 修改成功
  - “/export” 导出会话 → 查看 exports
