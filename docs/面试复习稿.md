# simbaAgent 面试复习稿（系统介绍模板）

这份文档用于你面对面试官时，按“业务价值 → 架构 → 关键设计 → 难点与取舍 → 可扩展”系统讲清楚项目。

## 1. 30 秒电梯陈述（必背）

我做了一个可演示的电商售后智能客服工作台：前端是 Vue 的 Inbox/Console，后端用 Express 提供 REST 和 SSE。

与普通“聊天机器人”不同的是，我把 Agent 的 **过程** 可视化了：后端不仅推 token 级回答，还推计划（plan_update）、工具调用日志（tool_call/tool_result）、以及需要人工确认的弹窗（need_confirm）。同时支持 MCP 扩展，能把会话导出到本地文件并写入 SQLite 作为审计。

## 2. 1 分钟版本（结构化讲法）

### 2.1 需求是什么

- 售后工作台需要“快”：客服要看到系统在做什么，而不是黑盒。
- 售后动作要“安全”：退货创建、改地址这种敏感操作必须二次确认。
- 要“可演示可扩展”：用 mock 数据跑通闭环，并能平滑替换成真实系统/更多工具。

### 2.2 我怎么设计

- 通信：用 SSE 做单向事件流，避免 WebSocket 复杂状态。
- 协议：把事件分成两类：
  - 文本：assistant_delta（逐字输出）
  - 过程：plan_update/tool_call/tool_result/need_confirm/...（可观测）
- Agent：LangChain + Tools，工具返回事实，模型负责组织回答。
- HITL：敏感动作先发 need_confirm，前端确认后再发一次请求执行。

## 3. 架构图（白板画法）

画 5 个框：

1) 前端 Console（Vue）
2) 后端 /api/chat/stream（Express）
3) SSE 事件管线（plan/tool/delta）
4) LangChain Agent（ChatOpenAI + Tools）
5) Tools/MCP（orders/shipment/kb + filesystem/sqlite）

箭头：

- FE → API：POST {conversationId, message, context}
- API → FE：SSE events（plan_update/tool_call/tool_result/assistant_delta/...）
- Agent → Tools：invoke
- Agent → MCP：导出/审计

## 4. 关键点深入（面试官常问的 6 个点）

### 4.1 为什么用 SSE，不用 WebSocket？

- 我这里主要是服务端持续推送，双向交互可以通过“下一次 HTTP 请求 + context”实现。
- SSE 更容易部署、调试、成本更低。

### 4.2 为什么不用 EventSource？

- EventSource 只支持 GET。
- 我需要 POST 并携带 body（conversationId/message/context），所以使用 fetch + ReadableStream 自解析 SSE。

### 4.3 如何保证回答可解释、可追溯？

- 所有事实来自工具：订单、物流、知识库条款。
- 工具调用过程通过 tool_call/tool_result 事件推给前端，直接展示给用户。
- 政策问答通过 kbSearch 命中 → 引用卡片（articleId/snippet）体现“回答依据”。

### 4.4 如何避免模型乱做敏感操作？

- 退货/改地址都走 Human-in-the-loop：need_confirm。
- 前端确认后才真正执行 createReturnRequest / modifyOrderAddress。

### 4.5 流式体验怎么做的？

- 后端：SSE + assistant_delta。
- 前端：fetch 拿到 res.body，用 getReader() 逐块读；buffer 拼接后交给 SSE 解析器。
- 预插空的 agent 消息作为“锚点”，每次 delta 只更新这一条消息的 content，避免列表抖动。

### 4.6 MCP 在这里解决了什么？

- filesystem：一键导出会话（面试演示很加分：可回放、可复盘）。
- sqlite：把请求记录写入数据库，支持 /audit 回看最近请求（体现审计意识）。

## 5. Demo 演示脚本（建议顺序）

### 5.1 查物流（展示工具链路）

输入：“我手机尾号3344，帮我查下物流”

你讲：

- plan_update：补齐订单信息 → 查订单详情 → 查轨迹 → 生成答复
- tool_call/tool_result：searchOrders / getOrderDetail / getShipmentTracking

### 5.2 退货（展示 HITL）

输入：“我要退货，质量问题”

你讲：

- 工具判断可退（getReturnEligibility）
- need_confirm 弹窗
- 点击确认后创建退货（createReturnRequest）

### 5.3 改地址（展示动作类工具）

输入：“我要改地址，新地址：广东省深圳市南山区xxx”

你讲：

- 订单状态拦截：已支付未发货才可改
- need_confirm → modifyOrderAddress → 回执

### 5.4 导出与审计（展示 MCP）

输入：“/export” 或 “/audit”

你讲：

- export_ready：返回导出文件路径
- sqlite_result：前端表格展示审计记录

## 6. 常见追问与回答素材

### Q1：这个项目如果上生产，你还要补什么？

- 真正的订单/物流/退款系统接入（替换 mock 数据）。
- 权限与风控：谁能改地址/退货；操作审计；幂等。
- RAG 升级：向量检索（pgvector/FAISS）替换关键词检索，保留事件协议。
- 可靠性：断线重连、SSE backpressure、并发与限流。

### Q2：你怎么衡量这个方案的效果？

- 体验指标：首包时间、首 token 时间、工具调用耗时。
- 业务指标：一次对话解决率、转人工率、误操作率（HITL 下降）。

## 7. 结尾 15 秒（收口）

这个项目的核心是把 Agent 从“只会回答”变成“可观测、可交互、可扩展”的工作台：过程事件让它可解释，确认闭环让它安全，MCP 让它能接入真实世界能力。
