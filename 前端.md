# simbaAgent 前端面试 Q\&A（基于当前仓库）

本文汇总面试官在只看前端实现时，最可能深挖的问题与可直接作答的要点，配合仓库内文件位置，便于你现场定位讲解。

***

## 1）SSE 为什么用 fetch + ReadableStream 而不是 EventSource？

- 需要 **POST** 请求并携带 body（`conversationId/message/context`），EventSource 只支持 GET。
- 用 `fetch` 可获得 `res.body.getReader()`，按字节增量解码 + 自定义解析更灵活。
- 参考：发送与读流在 [ConsolePage.vue](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L149-L191)，解析器在 [sse.js](file:///d:/code/study/simbaAgent/src/ui/pages/console/sse.js)。
- EventSource ：专为 SSE 设计，自动解析 + 自动重连，但只能 GET、控制力较弱
- fetch + ReadableStream ：更底层更灵活，支持 POST/body，自由度高，但需要你自己做解析、缓冲、错误处理与重连策略

## 2）前端如何解析 SSE？为什么要维护 buffer/rest？

- 解析规则：事件由 `event:`、`data:` 多行组成，**空行** **`\n\n`** 结束一个事件。
- 网络分片可能把一个事件切成多段，必须把新到文本 **拼接到 buffer**，只解析完整块，尾段放到 **rest** 等下次再拼。
- 参考实现：[parseSseTextChunk](file:///d:/code/study/simbaAgent/src/ui/pages/console/sse.js#L1-L25)。

## 3）为什么在 send() 里要先插入“空的 agent 消息”？

- 作为“流式承载点”，避免第一段 delta 到来时再插入节点引起首帧位移/滚动抖动。
- 只在同一条消息里 **更新 content**，虚拟 DOM 只做轻量 patch，滚动更稳定。
- 通过 `assistantMsgId` 精确定位，避免中途插入系统消息导致“写错对象”。
- 参考：[ConsolePage.vue（预插空消息 + 累积 delta）](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L128-L170)。

## 4）自动滚动如何实现？为什么有 locked？

- ChatPanel 监听 `[messages.length, streaming]` 变化，在 `nextTick` 后滚到底；用户手动上滑（离底部 > 240px）时 **locked=true**，不再强拉回底部，避免打扰阅读。
- 关键组合：父容器 `flex flex-col` + 子容器 `min-h-0 flex-1` + 滚动容器 `overflow-y-auto`。
- 参考：[ChatPanel.vue 模板与脚本](file:///d:/code/study/simbaAgent/src/ui/pages/console/ChatPanel.vue) 与 [ConsolePage.vue 左列布局](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L28-L35)。

## 5）为什么要用 nextTick 再做滚动/自适应高度？

- Vue 会把响应式变更批量异步渲染；`nextTick` 能保证“DOM 已经是最新”，此时读取 `scrollHeight`/设置 `scrollTop` 才准确。
- 参考：滚动在 [ChatPanel.vue](file:///d:/code/study/simbaAgent/src/ui/pages/console/ChatPanel.vue#L82-L95)，输入框自适应高度在 [Composer.vue](file:///d:/code/study/simbaAgent/src/ui/pages/console/Composer.vue#L26-L66)。

## 6）Sidebar 各区块如何由 SSE 事件驱动？

- Plan：`plan_update` → `planSteps`。
- 工具日志：`tool_call/tool_result` → `toolLogs`；同时解析出 `getOrderDetail/getShipmentTracking` 的结果更新 `order/shipments`。
- KB 引用卡片（RAG 演示）：`tool_result` 且 `toolName==='kbSearch'` → `kbHits`。
- MCP：`export_ready`（导出卡片）、`sqlite_result`（表格）。
- 参考：[ConsolePage.vue 事件分发](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L160-L191)、[Sidebar.vue](file:///d:/code/study/simbaAgent/src/ui/pages/console/Sidebar.vue)。

## 7）Tailwind 在本项目如何落地？为什么不用单独的 CSS/SCSS？

- 使用 **Tailwind v4 + Vite 插件**（无传统配置文件）：在 [vite.config.ts](file:///d:/code/study/simbaAgent/vite.config.ts#L1-L7) 注册 `@tailwindcss/vite`，全局样式 [main.css](file:///d:/code/study/simbaAgent/src/main.css) 只做基础底色，其他全部写在模板的 utility class 中。
- 优势：视觉语法统一（卡片套件：`rounded-2xl border bg-white shadow-sm`）、开发效率高、样式更可控。

## 8）为何页面骨架用 Grid + 内部用 Flex？

- Console 两列结构稳定，用 Grid 明确列宽（`md:grid-cols-[1fr_380px]`）。
- 左列内部需要“聊天区自适应 + 输入区固高”，Flex 更适合纵向分配剩余高度（`flex flex-col` + `min-h-0 flex-1`）。
- 参考：[ConsolePage.vue 主布局](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L28-L35)。

## 9）为什么 catch-all 路由写 `/:pathMatch(.*)*`？

- 这是 Vue Router 4 的**兜底路由**写法，匹配所有未知路径并重定向到 `/inbox`，防止 404 落空。
- 参考：[router.js](file:///d:/code/study/simbaAgent/src/ui/router.js#L11)。

## 10）弹窗为何用 `@click.stop` + 点击遮罩关闭？可访问性如何考虑？

- 遮罩层 `@click` 关闭；内容容器 `@click.stop` 阻止冒泡，避免点内部误关。
- `role="dialog" aria-modal="true"` 提供语义，提升无障碍支持。
- 参考：[ConfirmModal.vue](file:///d:/code/study/simbaAgent/src/ui/pages/console/ConfirmModal.vue#L1-L46)、[OrderChoiceModal.vue](file:///d:/code/study/simbaAgent/src/ui/pages/console/OrderChoiceModal.vue#L1-L49)。

## 11）为什么不在前端用全局状态管理（如 Pinia）？

- 当前规模下，状态皆由单页 Console 管理：SSE 事件集中在该页消费，拆成子组件以 props/emits 传递即可，复杂度可控。
- 如果引入多页共享/跨会话缓存/重放，则考虑 Pinia/Query 等。

## 12）错误处理策略是什么？

- SSE 启动失败：`!res.ok || !res.body` → 置 `error` 并 `streaming=false`。
- 事件级错误：收到 `error` 事件 → 在 ChatPanel 渲染错误条。
- 参考：[ConsolePage.vue 错误分支](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L170-L191)，[ChatPanel.vue 错误块](file:///d:/code/study/simbaAgent/src/ui/pages/console/ChatPanel.vue#L20-L24)。

## 13）如何避免“无限自增高度”的输入框？

- 自适应高度上限：`Math.min(180, el.scrollHeight)`；超出即出现内部滚动，避免顶起页面。
- 参考：[Composer.vue resize](file:///d:/code/study/simbaAgent/src/ui/pages/console/Composer.vue#L26-L40)。

## 14）如何保证“人机协同”体验自然？

- `need_choice/need_confirm` 事件分离出“选择/确认”逻辑；用户操作后再带 `context` 回流到 SSE 接口继续链路。
- 前端通过系统消息（role: "cs"）即时反馈“已选择/已取消/已确认”。
- 参考：回调处理在 [ConsolePage.vue](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L199-L241)。

## 15）列表滚动与性能：为什么不做虚拟列表？

- 当前消息量与工具日志条数可控（演示态），虚拟化收益不明显；首要目标是“可观测 + 正确滚动”。
- 若未来消息/日志规模增大，可对 ChatPanel/工具日志区块引入虚拟列表（如 `vue-virtual-scroller`）。

## 16）SSE 与组件拆分如何配合？

- SSE 聚合处理在 ConsolePage，**只下发纯数据**给子组件；子组件不感知 SSE 协议，仅渲染 props。
- 优点：协议变更（事件字段等）集中在一个文件改动，UI 结构稳定。

## 17）为什么 `whitespace-pre-wrap`？

- 聊天气泡要保留用户/Agent 的换行，同时遇到超长英文/URL 时不溢出容器（配合 `break-words` 的场景）。
- 参考：[ChatPanel.vue 气泡内容](file:///d:/code/study/simbaAgent/src/ui/pages/console/ChatPanel.vue#L13-L17)。

## 18）RAG 呈现为何放在 Sidebar 而不在气泡里？

- 把“检索证据（hits）”与“回答文本”分区展示：回答可读性更好，证据可独立复制、清除、延伸（未来可做展开原文/高亮）。
- 参考：[Sidebar.vue 的 KB 引用卡片](file:///d:/code/study/simbaAgent/src/ui/pages/console/Sidebar.vue#L166-L199)。

## 19）为何 `min-h-0` 必不可少？

- 在 `flex flex-col` 容器中，子元素要可滚动必须声明 `min-h-0`，否则高度收缩规则会导致滚动区拿不到剩余高度（经典坑点）。
- 参考：[ConsolePage 左列](file:///d:/code/study/simbaAgent/src/ui/pages/ConsolePage.vue#L28-L35)。

## 20）如何扩展/自测？

- 扩展：把 `kbSearch` 替换为向量检索；保留 `tool_result` 的 `hits` 结构与前端卡片即可无缝升级。
- 自测：在 Console 输入“政策/退货/物流”等问题，观察工具日志与引用卡片、Plan、聊天流式是否同步更新。
